# 割草机时间戳同步噪声优化方案

# 1. 原方案

[ 割草机时间戳同步方案设计](https://roborock.feishu.cn/docx/FmeadwBfvodJ7GxXFQKc6r56nUe)

# 2. 更新方案

同步初始化：

1、pps信号后，25ms（或其他阈值）内没有报文，放弃此次初始化。

2、pps信号中断到来 得到ap的 tick1； 获取 25ms内的报文，得到UTC1， 初始化 offset1 = UTC1 - tick1  。  完成初始化。



后续的每次同步：

1、pps信号后，25ms内 没有报文数据可拿，放弃此次同步。

2、pps信号后，拿到了AP当时的 tick\_i； 获取了RTC报文，得到了UTC\_i   更新offset = UTC\_i - offset1 - Tick\_i



# 3. 新方案原理、异常状态分析

第i个周期AP收到pps中断，AP读取紧接着的报文，得到第i个周期的UTC时间，记为UTCi
第i个周期AP收到pps中断，AP记录自己的时间戳，记为Ticki
offset = UTCi - offset1 - Ticki （offset1的计算方式，与原文档保持一致。[ 割草机时间戳同步方案设计](https://roborock.feishu.cn/docx/FmeadwBfvodJ7GxXFQKc6r56nUe)）
第i个周期到第i+1个周期的AP时间t = Tick(当前AP时间戳) + offset

1s一个pps信号，1s有1个RTK报文，RTK报文晚于pps中断约20ms，与UTC(i-1)的差>995ms且<1005ms的是对应pps信号的UTCi(这里不严格卡1s，防止卫星报文跳动。这里限制的原因是防止丢报文。)

为防止AP收到pps中断到AP收到RTK报文之间再收到一个噪声PPS信号，下面把i+1'周期当做噪声，i+1周期当作真信号分析

收到i+1'时刻的PPS假信号，如果25ms内没有收到报文，这个假PPS信号不进行同步。
如果25ms内收到报文，计算offset = UTC(i+1') - offset1 - Tick(i+1')
AP校正后的时间戳t = Tick(当前AP时间戳) + offset.&#x20;
这里的UTC(i+1')无法与PPS假信号对齐，因为它不是真脉冲。UTC(i+1')其实就是UTC(i+1)，因为1s最多1个报文。offset = UTC(i+1') - offset1 - Tick(i+1')，这里Tick(i+1')是AP收到假PPS的本地时间戳，所以这次offset可能有点波动，这个波动小于25ms，且这次波动的影响持续最多1s。但是在i+2时刻，可以正常同步，计算offset = UTC(i+2) - offset1 - Tick(i+2)，继续计算t就恢复准确AP时间戳。

再考虑第一个PPS信号就是噪声(对应AP时间Tick1')，
若25ms内没收到UTC1，则这个PPS信号不启动时间戳同步。
若25ms内收到UTC1，则offset1 = UTC1 - Tick1'是错误的，但这个错误对于t的计算就是一个常量。这个常量偏差不影响整个系统工作。
