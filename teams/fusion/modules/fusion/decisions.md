## D-001 传感器组合选型【已定案】

**背景**
室外割草机场景：空旷、草皮重复纹理、存在遮挡/打滑/漂移干扰

**选定方案**
RTK + 双目视觉 VIO + 轮速计 + 陀螺仪 + 加速度计

**理由**

- RTK：空旷场景厘米级精度，提供绝对位置
- 双目 VIO：草皮丰富纹理支持局部 6DOF 相对位姿，RTK 遮挡时短时过渡
- 轮速计+陀螺仪+加速度计：短时相对位姿预测（方差设大，防打滑/漂移累积）
- 极端情况（多传感器均失效）：上报报警信息，导航调整路径

## D-002 选用 ESKF 框架作为多传感器融合核心【已定案】

**背景**
需要一个能融合 RTK（位置观测）、VIO（局部位姿观测）、轮速+IMU（预测）的统一滤波框架

**选定方案**
ESKF（Error-State Kalman Filter）

**理由**

- KF：仅适用线性系统
- EKF：对非线性系统全局线性化，精度受限
- ESKF：名义状态保持非线性更新，只对误差状态（小量）线性化 → 数值更稳定、计算效率更高；适合旋转、位姿等高维非线性系统

**来源** | 002_算法文档/006_ESKF算法

## D-008 融合 RTK + VIO 逻辑 v1.0【已定案（已迭代至 v1.1）】

**背景**
ESKF 框架中需要决策何时接入视觉观测、何时信任 RTK 观测

**选定方案**

- `check_insert_entry()`：观测是否加入队列
- `check_update_entry()`：观测是否可用于更新
- 视觉使用策略：RTK 固定解时不使用视觉；非固定解超时后使用视觉；接入视觉时 RTK 与视觉起点对齐

**优化记录 v2（2025-07-03）**

- VIO R 矩阵设为 1e-8/1e-6/1e-4（置信度调节）
- 星数 <21 颗的固定解不参与更新
- 结论：视觉累计误差优于 IMU/Gyro

**参数背景（2025-10-11，林子越）**

- RTK 位置观测方差约 **1e-3 量级**（标准差），位置输出基本跟随 RTK
- 但 RTK 仅提供位置观测，**航向角不经过滤波器直接修正**；yaw 收敛依赖 RTK-odo 对齐线程（见 D-010）；这是 P-006（yaw 强制替换而非进滤波器）问题的根因背景

## D-009 融合 RTK + VIO 逻辑 v1.1【已定案】

**背景**
v1.0 在 RTK 4/5/4 来回跳变时处理不足；假固定识别能力弱

**选定方案**

1. 加入 RTK 位置方差/速度方差判断假固定（优先于后续判断）
2. 相邻 RTK 跳变 >0.2m 或状态转换（4↔5）时，改用 VIO 递推
3. VIO 检测 RTK 一致性：维护 20s 双队列，首尾位移偏差 <10% 才切换回 RTK 固定解
4. 静止状态下不响应 RTK 跳变

**风险**
5→4 且 RTK 为真固定解时，会多递推 20s

> **后续迭代**：v1.1 队列机制在 2026-01 进一步细化为跳变后倒退窗口 + 收尾向量夹角/长度双维度判断，见 D-037（假固定处理 4.0/5.0）。

## D-010 RTK-odom 对齐线程设计【已定案】

**背景**
浮点→固定解、视觉→固定解等特殊切换时，融合定位 yaw 收敛慢

**选定方案**
独立对齐线程，一直做 odom 与 RTK 轨迹 SVD 对齐：

- 队列 fixed_size=10，RTK 数据均为固定解
- 质量检查：帧间行驶角度 <5deg，累计 RTK 长度 <1m（防固定解异常）
- 残差验证：平均角度误差 <5deg 认为 yaw 有效
- 时间戳检查：get_pose 与计算 yaw 时间差 <50ms 才使用

**后续改进方向**

- 滑动窗口替代 fixed_size；优化质量检查标准；yaw 作为观测进滤波器而非强制更改状态

**来源** | 002_算法文档/009_Rtk odom初始化线程

## D-011 非搬动重定位（融合定位）方案【已定案】

**背景**
RTK 非固定解 + vslam tracking lost 时，需要 SLAM 主动发起重定位而非等待外部触发

**选定方案**

1. vslam 维护队列，tracking lost 时提供 `T_wv0`（丢失时刻位姿）和 `t0`
2. 融合维护等长惯导队列，收到后算出当前位姿 `T_wf1`
3. 融合通知导航，导航转 180°回 RTK 良好区域（`T_wf2`）
4. 融合初始化 vslam（位姿 `T_wf1`，速度≈0）
5. vslam 尝试重定位；成功则正常割草；失败则依赖惯导沿原路返回，或走 2×2 矩形搜索

**来源** | 002_算法文档/008_重定位/002_非搬动重定位（融合定位）方案

## D-015 MCU 时间同步方案【已定案】

**背景**
三处理器（MMCU/VMCU/AP），MMCU 以 10ms 读编码器，VMCU 以 20ms 上传 AP，两 MCU 无法直接时间同步

**选定方案**
MMCU 不打时间戳；VMCU 在收到左轮数据中断时打时间戳 `t2`（左右轮共用），消除 MMCU 内部 2ms 间隔影响。VMCU 无论是否凑齐 2 包都立即上传。

**放弃方案**
MMCU+VMCU 时间同步（修改量大）

## D-018 nRTK 软件架构总体方案【已定案】

**背景**
产品需要支持主机通过 4G 连接 CORS 服务（网络RTK）替代 LoRa 基站方案，需要设计账号管理、坐标转换和多层软件交互。

**选定方案**

- 【中间层】调用 AP 接口关闭 LoRa，init_nrtk() 建立 NTrip 服务；定时器每 10s 发送 GGA 报文维持虚拟站；AGRICA 解析输出 BLH 坐标给定位
- 【定位】nRTK 模式下获取经纬高后转 ENU 坐标；建图第一个 RTK=4 点为原点；重启读取地图参考点
- 【导航】建图时在导航地图中新增基准原点（BLH 坐标系），其余接口不变
- 【账号管理】账号取还 + TTL 检验 + 异常更换；IoT 平台：账号池绑定 SN、设备离线 5min 后主动解绑、1h 心跳保活
- 【AP-Wlan Manager】任务期间 WiFi 信号不佳直接切换 4G，不再切回

**理由**
主机 nRTK 相比 1+1 RTK 减少一个基站模组，降低成本；通过 CORS 服务获取差分数据，适合无固定基站场景

**来源** | inbox/011_软件架构/010_nRTK技术方案/nRTK技术方案.md

## D-019 基站RTK / 主机nRTK / 桩nRTK 三模式切换设计（Gaia）【已定案】

**背景**
Gaia 产品需要支持三种 RTK 模式（1+1 基站RTK、主机nRTK、桩nRTK）之间的任意切换，且切换后地图需对准或复用

**选定方案**


| 切换方向            | 地图处理                    |
| --------------- | ----------------------- |
| 基站RTK → 主/桩nRTK | 需地图对准（坐标系变换）            |
| 主nRTK ↔ 桩nRTK   | 不需地图对准（同为 nRTK，绝对坐标系相同） |
| 主/桩nRTK → 基站RTK | 需地图对准                   |


- 【APP Proxy】透传切换指令、负责账号取还
- 【状态机】维护 RTK/nRTK 双状态机，处理故障码，调接口开/关 LoRa
- 【中间层】下发 set_nrtk / set_rtk，账号下发，GNGGA/AGRIC 解析
- 【AP/MCU】半双工串口模式调整，接收并切换通信模式
- 【导航】切换前暂存 ENU 坐标，切换后地图坐标转换，删地图时判断当前模式仅删当前模式地图
- 桩nRTK 新增：桩端 esp32 进行 Ntrip 鉴权、自组 GNGGA（2s 频率）、LoRa 传 RTCM 给主机；主机同步桩坐标到桩端

**理由**
主/桩 nRTK 共享同一 CORS 绝对坐标参考，切换时无需重校准；基站RTK 与 nRTK 坐标原点不同，须坐标转换

## D-020 nRTK Auto 模式智能切换策略（Gaia）【已定案】

**背景**
用户不希望手动管理主机nRTK/桩nRTK切换，需要系统根据 LoRa 信号和网络状态自动决策

**选定方案**

- 设置 Auto 模式时先进行桩 LoRa 配对；初始若网络可用 → 主机nRTK，若不可用 → 桩nRTK
- 建图时：网络可用强制切主机nRTK
- 割草时切换策略：
  - 切桩nRTK 条件：桩在线 + LoRa 信号正常（开始割草时检测）
  - 回主机nRTK 条件：LoRa 中断（15s 内无心跳包）/ LoRa 即将断开（15s SNR均值<2dB）/ RTCM 中断（>30s 无差分包） 且主机已联网
  - 切回桩nRTK 条件：LoRa 稳定（15s SNR均值>4dB）+ 云端查询桩在线

**理由**
桩nRTK LoRa 无网络延时，信号稳定时优先使用；主机nRTK 作为桩nRTK 不可用时的兜底，保证定位连续性

## D-021 RTK 非固定解→固定解恢复时姿态收敛方案选型【已定案】

**背景**
Bug#391664：400 平地图弓字避障出界。根因：从 RTK 非固定解（使用视觉）恢复到 RTK 固定解时，姿态无法快速收敛，导致定位偏差出界。

**五种候选方案对比**


| 方案      | 思路                                                  | 评估                    |
| ------- | --------------------------------------------------- | --------------------- |
| 方案一     | 修改更新方程，将 odo/RTK 速度作为新观测加快姿态收敛                      | 效果不好（实验验证）            |
| 方案二     | 使用完整 IMU 递推方程 + 方案一的速度观测                            | 方案一效果差则此方案类似          |
| 方案三     | 抛弃之前姿态，直接将航向设为 RTK 固定解速度方向                          | RTK 速度经验证可信（倒车/前进均准确） |
| 方案四     | 重新进行 SLAM 初始对准                                      | 延迟大，体验差               |
| **方案五** | 后台线程持续做 odo-RTK 轨迹对齐，切换时以 RTK 速度方向为初值，后台对齐完成后替换 yaw | **选定**                |


**选定方案（方案五）：轨迹对齐**

单独开后台线程持续计算 odo 与 RTK 的对准 yaw。当从非固定解切换到固定解时：

1. 实时航向初值设为 RTK 速度方向
2. 后台线程完成 odo-RTK 对准计算后，以计算结果的 yaw 角替代状态中的 yaw

融合速度处理方案五：轨迹对齐设计

**理由**
方案五兼顾快速响应（RTK 速度方向初值）和精度（后台精确对齐），避免了方案三在低速/倒车场景下速度方向不稳定的风险。

## D-022 nRTK 软件实现细节（中间层/定位/导航/账号管理）【已定案】

**背景**
基于产品需求（网络RTK PRD），分解 nRTK 各软件层的具体实现方案（V0.2 初版 → V1.1 最终版）。与 D-018 不同：D-018 描述架构总体方案，本条记录各层的实现细节和接口约定。

## D-025 通道外假固定判断策略（1.0 → 2.0）【已定案】

**背景**
RTK 在树下、屋檐下、通道外等受遮挡区域会出现"假固定解"——显示 Fixed 状态但实际误差大。需区分通道内/外状态，对假固定解做屏蔽并切换到 VIO 辅助。

**选定方案（2.0）**

通道进入/退出判断条件：


|     | 进入通道内                       | 退出通道（回到通道外）                                                   |
| --- | --------------------------- | ------------------------------------------------------------- |
| 通道内 | 浮点解 / 3d跳变>10cm / 单帧std超阈值  | 固定解 && 速度>0.03 && 连续5帧                                        |
| 通道外 | 浮点解 / 2d跳变>15cm / 3d跳变>20cm | 固定解 && 速度std<0.2 && 位置std<0.02 && 速度>0.06 && 相邻两帧rtk2d距离<10cm |


**2.0 新增：** 兜底逻辑——整个区域 RTK 受影响、无符合条件固定解约 600 帧时强制恢复；增加单帧角度纠正航向角（先判速度std）；1.0 2d 跳变阈值从 20cm 收紧到 15cm。

**理由**

- 1.0 退出通道条件过宽，树下/屋檐下假固定解仍被接受，导致定位跳变
- 2.0 增加 3d 跳变条件和速度 std 限制，减少误判

## D-027 VIO 对齐 RTK 方案（Pose Graph 优化）【研究中】

**背景**
RTK 信号恢复后，VIO 的累积漂移需要通过 RTK 固定解约束来校正。传统方法直接跳变；Pose Graph 方法通过优化 VIO 历史轨迹实现平滑对齐。

**选定方案**

约束建模（三类）：

1. RTK 与 VIO 帧时间对齐约束
2. VIO 帧间位姿连续性约束
3. VIO 帧间尺度与方向一致性约束（变换后相对位置尽可能不变）

关键结论：**加入尺度相似性约束**——只有靠近 RTK 固定解一定范围内的 VIO pose 会被优化，整体轨迹变化小；无该约束则全局轨迹变化随机，不可控。

**当前状态**：基本流程打通，优化权重（角度方向）尚未调好；多个 bug 数据已验证效果有改善（Bug#487694, Bug#470159, Bug#482646, Bug#478129 等）。

## D-028 卡困检测细节逻辑（odo/gyro 旋转打滑 + RTK 前进/后退打滑）【已定案】

**背景**
机器割草时遇到障碍或草地打滑，需检测打滑事件并向上层（导航/调度）上报，触发脱困。

**选定方案**

两类检测：

**1. 基于 odo & gyro 旋转打滑检测**

- 滑窗内 odo 累积角度与 gyro 累积角度差超阈值触发
- 误报防护：odo 绝对角度需大于最小阈值（防静止时 gyro 漂移）；两者差值须超过正常旋转打滑基准
- 滑窗重启：任一传感器前后时间 gap 过大时重启

**2. 基于 odo & RTK 固定解前进/后退打滑检测**

- 滑窗内 odo 位移与 RTK 位移差超阈值触发
- 误报防护：时间戳对齐、gyro 旋转角度小于阈值（防 U 型轨迹）、RTK 移动量小于阈值

打滑方向编码：rotate_slip / move_slip 各 -1（右/后退）/ 0（不打滑）/ 1（左/前进）

**参数背景（2025-11-26，李哲）**

- 斜坡受困场景：gyro-odo 旋转检测阈值讨论提升到 **140°**（或改用「接近目标点」判断逻辑）
- **车裂检测**：前轮 x 方向速度 > 后轮速度超过阈值持续 **9s**，报受困（与 D-029 M=9s 窗口对应）
- 曹丽娜建议：「机器是否被困住」的判断逻辑应在导航侧做（最终决议见 D-029）

## D-030 RTK 定位与重定位系统设计规格（含阴影区策略）【已定案】

**背景**
割草机产品需要在不同工作状态（暂停/遥控/出桩/割草中/被搬动/信号丢失）下维持准确定位，并在搬动或信号丢失后有效重定位。本文档为产品级设计规格（PM 侧 @小倩），对应 G-004 飞书 wiki 本地化版本。

**选定方案**

定位基础逻辑：

- 主机正常暂停且无搬动/主动开关机 → 默认当前位置为历史位置
- 遥控割草中 → 定位模块保持更新
- 在桩触发任务 → 默认基站位置为历史位置（针对无 RTK 信号充电基站位置场景）
- 定位初始化：出桩动作中完成；遥控建图场景在用户遥控过程中完成

融合定位方案：RTK 天线 + 机器人 VSLAM + GPS + IMU + 里程计数据融合精确定位；NRTK（欧区）/ 1+1 RTK（美区）并行做

RTK 信号质量热力图（三档）：

- 良好：固定解，搜星数 > 阈值
- 一般：固定解，搜星数 < 阈值
- 差：浮点解
- 阴影区判定标准：长宽 > 2m×2m 认定为阴影区

阴影区处理（RTK 信号丢失时）：

- 记录最后一次信号良好的位置，结合整机 IMU 进行当前位置粗定位
- 向大致开阔区域/局部最高点前进，同时摄像头边界判断确保不出界
- 超时 5min 停机报错

重定位动作（两驱 & 四驱）：

- 以机身当前点直行 1m 后，以机身为中心走 2m×2m 方格

重定位触发时机：

1. 搬动后 / 基站外重新开机后触发新任务
2. 待机回充前无定位，再尝试一次（无法多次尝试）

出桩定位：

- 判断是否触发过搬动/关机，若有：出桩过程中 RTK + 双目重定位确认桩位置变化
- 无变化：启动割草；有变化（地图内）：自动更新桩点；有变化（地图外 >1m）：回桩停机提醒；有变化（≤1m）：更新桩位置及上桩通道；定位失败：回桩报失败

1. 遥控出桩 → 行进至 RTK 固定解区域

## D-031 搬动状态下定位模式切换方案比选【待确认】

**背景**
需求：机器被轻微搬动时不进行重定位，在搬动中仍能显示位置（不显示轨迹/方向）。需要在保留定位能力的同时减少不必要的重定位次数。

**两种候选方案**

**方案一（利用 pause 切模式）**：pause 时切换为简化位置 + 互补滤波姿态；slam 状态切换到 2（简化），恢复后通过状态 0/1 通知导航是否需要重定位。

- 优点：能 cover 大多数「先暂停再搬动」场景（占绝大多数）；不需要 slam 独立做抬起检测
- 缺点：运动时搬动场景下抬起检测时间长（驱动 0.2s + 导航 2s），姿态可能错误更新；RTK 抬起时轮子未停可能发生打滑，位置被推偏

**方案二（slam 自检抬起状态）**：不依赖 pause 指令，slam 自己通过加速度阈值检测抬起/放下状态，切换为简化位置 + 互补滤波姿态。

- 优点：运动时搬动也能 cover；可立即报抬起，不用等 pause 的 2s 延迟
- 缺点：抬起检测阈值难定（短则误检，长则姿态错误更新时间长）；误检时影响脱困流程；需要与脱困状态机梳理清楚关系

**方案二具体检测逻辑（2026-01-28，李哲）**

判断为「抱起」的条件（满足任一）：
1. IMU 姿态倾斜 >45°
2. 轮子全部 lift + IMU 加速度超阈值
3. 机器人高度 >0.5m

处理流程：
- 抱起 → 立即暂停（刀盘停、轮子停）
- 卡住 → 进入脱困逻辑，8s 后仍异常则报错

**方案对比**


|     | 暂停情况下搬动（大多数情况）     | 运动时搬动             | 运动时无搬动      | 对重定位次数影响 |
| --- | ------------------ | ----------------- | ----------- | -------- |
| 方案一 | 能 cover            | 抬起检测时间长，姿态错误更新时间长 | 不变动         | 减少       |
| 方案二 | 能 cover，需额外自己做抬起检测 | 能 cover           | 可能误检，会影响脱困等 | 不确定      |


**当前状态**：两种方案均已分析，未给出最终选型结论（见 Q-006）。

## D-032 导航-SLAM 重定位三场景接口设计【已定案】

**背景**
割草机重定位逻辑涉及三个触发场景，SLAM 与导航之间通过 protobuf 消息交互。各场景发起方、SLAM/导航行为、结束判定均不同，需统一接口规范。

**三场景分工**


| 场景                  | 发起方            | 导航行为            | 结束判定                         |
| ------------------- | -------------- | --------------- | ---------------------------- |
| 割草中定位差（RTK + 视觉均丢失） | SLAM 主动通知导航    | 速度减半，纯 odo 递推   | SLAM 判断过去 3s RTK 固定解比例 > 95% |
| 机器被搬动               | 导航检测搬动后通知 SLAM | 走 2m×2m 方格      | SLAM pose.init 0→1           |
| 其他（基站外重启等）          | 导航主动 reset     | 复用 reset_cmd 流程 | —                            |


**接口定义（common_protobuf/proto/common/SlamMsg.proto）**

场景一（SLAM → 导航）：

- `SlamToNavMsg::Type::RELOCATE_BAD_LOCATION = 2`，`relocate_cmd = 3`（1=开始，0=结束）
- 状态机交互：建图模式→报错；割草模式→忽略（进入主动重定位状态）

场景二（导航 → SLAM）：

- `NavToSlamMsg::Type::RELOCATE_MOVED = 10`
- 重定位成功条件：SLAM 找到 RTK 固定解 + 重新初始对准成功

场景三：复用已有 `NavToSlamMsg::reset_cmd`，无新接口。

**产品超时策略**：定位长时间无法找回，5min 停机报错。

**待确认** 见 Q-007、Q-008。

## D-034 轨迹横向抖动降低方案（residual 限幅 + NHC + RTK 方差）【已定案】

**背景**
直线段割草出现左右摇摆（"超级大直线左摇右摆"），根因是定位 pose 横向漂移影响导航控制，需要从源头抑制融合输出的横向噪声。

**选定方案**
三项组合措施：

1. **residual 限幅**：对 ESKF 更新残差做截断，防止单次大残差引入跳变
2. **NHC（Non-Holonomic Constraint）约束**：利用地面车辆侧向/垂直速度为零的约束，作为辅助观测抑制横向漂移
3. **增大 RTK 观测方差**：适当放宽 RTK 位置观测置信度，减少 RTK 偶发噪声对融合状态的横向拉扯

**目标精度**：横向误差 ±2.5cm

## D-035 重定位失败后直接报错、不做 RTK 重启【已定案】

**背景**
机器重定位失败时，存在通过重启 RTK 模组来重新获取固定解的备选方案。需要评估此方案风险。

**风险评估结论**（2025-11-13，李哲 / 施敏杰）


| 风险级别    | 问题                              |
| ------- | ------------------------------- |
| 不严重     | 航向变化（重定位中无所谓）；位置跳变（不会发生）；温启动耗时长 |
| 中等      | 需要重新做时间同步                       |
| **高风险** | 重启后假固定解是否可信（无法保证）               |


**选定方案**：重定位失败后直接报错，不做 RTK 重启。

**理由**：重启后假固定解可信度无法保证，风险不可控；产品策略已定为超时 5min 停机报错（见 D-030）。

**来源** | 群聊讨论 2025-11-13（李哲 / 施敏杰）

## D-037 RTK 假固定解 Rework 机制（跳变后倒退窗口 + 队列对比）【已定案】

**背景**
D-009 v1.1 建立的双队列机制在 RTK 跳变后处理不够精细——切回 RTK 的判断标准单一（仅看位移偏差 <10%），对浮点/固定解切换以及融合队列利用不足。2026-01 讨论迭代机制（对应假固定处理 4.0/5.0，见 G-033/G-032）。

**选定方案（2026-01-15~16）**

核心思路：
1. **跳变倒退窗口**：发现 RTK 跳变后，向前追溯一段时间，在此时间窗口内不信任 RTK
2. **队列对比**：同时累积 RTK 队列和 fusion 队列，对比两者轨迹一致性来判断 RTK 是否恢复
3. **切回条件**：首尾向量**夹角 + 长度**两个维度综合判断（替代原单一位移偏差阈值）；若偏差 >10%，清空重新统计；否则切回 RTK
4. **融合队列来源**：直接使用 odom，当 VIO lost 时降级处理

特殊情况处理（曹丽娜）：
- 机器**长时间暂停** → 清空队列，防止历史数据污染
- 浮点解和固定解切换 → 进入单独的切换逻辑，不混入队列对比

**RTK 失效四类场景（2026-01-15，曹丽娜）**

| 场景 | 描述 |
|---|---|
| 缓慢漂走后跳回 | RTK 缓慢漂移到错误位置，随后突然跳回 |
| 跳变失效后跳回 | RTK 突然跳变到错误位置，随后又突然跳回 |
| 跳变失效缓慢飘回 | RTK 突然跳变到错误位置，随后缓慢漂移回来 |
| 静止时跳出短暂又回 | 机器静止时 RTK 短暂跳出一段时间后回来 |

**来源** | 群聊讨论 2026-01-15~16；假固定处理 4.0（G-033）/ 5.0（G-032）

## D-038 夜间 VIO 关闭策略【已定案】

**背景**
VIO（视觉里程计）依赖图像纹理，夜间光照不足时无法正常工作，影响 RTK 阴影区定位能力。

**选定方案**
- 夜间**关闭 VIO**，不尝试视觉更新
- 通过**补光灯是否开启**作为夜间模式的确认条件（补光灯开 = 夜间模式）
- RTK 阴影区割草夜间**降级为纯 odom 递推**（失去视觉辅助兜底能力）

**已落地**：视觉黑夜 dark 功能，2026-04-02 合入（茹毅超）

## D-039 建图 PGO 优化方案（VIO 帧间约束 + RTK 先验）【研究中】

**背景**
建图过程中 RTK 信号偶尔差，导致建图轨迹漂移，事后无法修正。需要在建图期间或恢复信号后进行全局轨迹优化。

**方案（2026-03-11~12）**
- 建图过程中持续做 **Pose Graph Optimization (PGO)**
  - VIO 作为帧间相对位姿约束
  - RTK 固定解作为绝对位置先验约束
- RTK 信号丢失超时后**继续行走**，不停机；等 RTK 恢复后对整段轨迹做全局 PGO 优化

**配套细化措施**：
- 建图过程中**收严假固定解判定**（防止假固定解污染先验）
- 第一个区域起点附近**必须保证是 RTK 固定解**
- VIO 行驶距离 >20m 或时间 >1min，记录标志位（表示需要 PGO）
- 有标志位时：若终点附近固定解积累不足，引导用户**继续遥控**至 RTK 良好区域再完成

**当前状态**：方案讨论完成，实现状态待确认（见 Q-009 相关方向）

**来源** | 群聊讨论 2026-03-11~12
