# RTK割草机搬动

# 一、需求

1. 减少重定位，轻微搬动情况下不进行重定位
2. 在搬动中显示位置（可以不显示轨迹，不管方向）

# 二、情况分析

- 方案一：如果使用pause作为切换定位模式（切之前是正常ESKF，且之后是简化位置+互补滤波姿态）的指令，可能有两个问题对应的解决方案：
  1. 由于大多数常见情况下，机器都是先暂停，再搬动，最后恢复，所以暂停期间切换此模式可以cover大部分抬起情况；仍然由导航进行抬起检测
  2. 暂停期间，slam状态切换为2，在恢复后，通过转换状态为0或1，通知导航是否进行重定位
  3. 普通pause之后还有正常的运动（刹车）
    1. 就把这个情况当成抱起处理，也可以减少重定位
  4. pause之前就被抬起（需要lift一段时间才判定pause）
    1. RTK
      1. 抬起轮子没停，发生“打滑”，位置可能被推偏，收到pause指令时回复正常位置
      2. 抬起检测时间比较长（驱动0.2s+导航2s），从抬起到暂停期间，机器的姿态可能被更新错误
    2. 激光
      1. 抬起轮子没停，发生“打滑”，激光定位结果正确，融合位置可能被推偏，收到pause指令时回复正常位置
      2. 抱起可能遮挡雷达，
        1. 如果能够稳定识别遮挡导致的丢定位，可以通过主动重定位解决。
        2. 如果没法识别，雷达项目还是抬起后进行重定位
- 方案二：不用pause，就是之前那版的方案，需要新增一个状态：抬起/放下，进行切换。
  1. 在导航抬起检测不变动的情况下，slam检测自己的抬起状态，抬起后简化位置+互补滤波姿态
  2. 暂停时也需要计算
  3. 缺点：这个抬起放下需要跟脱困梳理清楚关系，
    1. 假设报抬起，然后又脱困，会用简化位置进行脱困，
      1. 如果是RTK固定解，没有影响
      2. 如果是RTK非固定解，可能会遇到问题（这个时候使用vio之前没有考虑过，因为视角问题，可能很容易就触发vio reset了）
    2. 抬起检测阈值比较难定
      1. 检测时间太短，误检较多
      2. 检测时间太长，姿态错误更新的时间长
  4. 优点：抬起放下不用跟pause等2s之间产生关系。
    1. 假设加速度阈值没法区分颠簸和抬起，这个方案可以立即报抬起，不用等


|     | 暂停情况下搬动（大多数情况）     | 运动时搬动              | 运动时无搬动      | 对重定位次数影响 |
| --- | ------------------ | ------------------ | ----------- | -------- |
| 方案一 | 能cover             | 抬起检测时间长，姿态错误更新的时间长 | 不变动         | 减少       |
| 方案二 | 能cover，需要额外自己做抬起检测 | 能cover             | 可能误检，会影响脱困等 | 不确定      |
