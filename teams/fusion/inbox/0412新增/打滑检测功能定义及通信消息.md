# 打滑检测功能定义及通信消息

# 1. v0.1

## 1.1 功能定义

1. 消息交互的方式，是打滑期间持续发还是状态改变时发一次。

   1. 状态改变时发一次IPC消息

      1. 好处是消息的发布只有一次。

      2. 麻烦的地方是需要考虑丢包情况，要用IPC消息，slam还要考虑丢报了再持续发，比较麻烦。

   2. 打滑期间持续发广播消息（prefer）

      1. 好处是简单直接。坏处是打滑期间的消息会多一点。（应该还好）

      2. 用广播消息即可，定一个频率。

      3. 打滑了发布打滑消息

         1. 打滑类型：单轮打滑/直行打滑

         2. 打滑时间：打滑发生的时刻

   3. 目前比较偏向b方案。我们先定一个频率，10Hz。

2. 打滑检测的功能

   1. **基于odo & gyro的旋转打滑检测**

      1. 打滑的定义

         1. M秒内联系w\_gyro - w\_wheel，大于N rad/s

         2. 目前vslam算法内部阈值为M=1.4s， N=0.15（1.4s内odogyro的角速度积分差大于12.03°），这个对于脱困来说太容易触发了。

         3. 算法准备将M设置为9s（机器旋转按40°/s转360°的时间），N=0.15不变。对应9s内odogyro的角速度积分差大于77.3°。

            1. 考虑到割草机正常情况下旋转打滑比较严重，必须hold一般的弓字割草旋转。所以M要大于转180°。

   2. **基于odo & RTK固定解的前进/后退打滑检测**

      1. M秒内联系move\_odo - move\_RTK，大于N米

## 1.2 消息定义（广播消息）

```plain&#x20;text
// message id: eRRMsgType_SlipDetected

typedef struct rr_msg_slip_info {
    uint32_t timestamp; // 正常都是uint64_t，但是实际上用不到那么多位表示时间
    int8_t rotate_slip;
    int8_t move_slip;
} rr_msg_slip_info_t;
```

timestamp // 判断窗口（9秒）内最后一个数据的时间戳，单位毫秒

rotate\_slip // 打滑与方向标识，-1为向右打滑（odo想向右转，但没转动，对应左轮打滑更严重，右轮卡困更严重）， 0为不打滑，1为向左打滑

move\_slip // 打滑与方向标识，-1为后退打滑， 0为不打滑，1为前进打滑
