# 定位组管理驾驶舱 — 设计文档

> 本文档说明整套体系的设计思路、文件结构、使用规范和维护原则。
> 所有协作者（包括 AI）在处理本工作区文件前，应先阅读本文档。

---

## 一、背景与目标

本工作区是定位组 TL 的**个人管理驾驶舱**，用于：

- 建立并持续维护对组织、任务、风险、技术、协同、人员六个维度的完整认知
- 作为日常管理的信息基座，而非对外汇报材料
- 支持「先收集原始资料 → 再提炼结构化信息」的工作流

---

## 二、整体结构

```
LocalizationTeam/
│
├── DESIGN.md                    ← 本文件：体系设计说明
├── README.md                    ← 总索引：每个文件是什么、怎么用
├── inbox/                       ← 全局原始资料（跨组会议记录、上级要求等）
│
├── overview/                    ← 横向管控视角（高频使用层）
│   ├── org.md                   ← 组织全貌 + 运行机制 + 对上接口
│   ├── tasks.md                 ← 全组任务汇总 + 关键路径
│   ├── risks.md                 ← 全组风险汇总 + 升级状态
│   └── collab.md                ← 跨组/跨项目协同 + 卡点 + 拍板人
│
├── teams/                       ← 小组纵深视角
│   ├── laser/                   ← 多线激光组
│   │   ├── tech.md
│   │   ├── people.md
│   │   └── inbox/
│   ├── fusion/                  ← 融合定位组
│   │   ├── tech.md
│   │   ├── people.md
│   │   └── inbox/
│   └── vision/                  ← 视觉定位组
│       ├── tech.md
│       ├── people.md
│       └── inbox/
│
└── projects/                    ← 项目横切视角（4～6 个项目平铺）
    └── <project-name>/
        ├── overview.md          ← 项目背景、定位组职责、关键节点
        ├── status.md            ← 当前进展、风险、跨组依赖
        └── inbox/               ← 该项目原始资料
```

---

## 三、两个主视角的关系

| 视角 | 目录 | 回答的问题 |
|------|------|-----------|
| 小组纵深 | `teams/` | 这个组在做什么？技术方向对不对？人员状态怎样？ |
| 项目横切 | `projects/` | 这个项目定位组负责什么？现在到什么阶段？风险在哪？ |
| 全局管控 | `overview/` | 整体有哪些风险？关键路径卡在哪？协同问题谁来拍板？ |

同一条信息可以被两个视角引用，但**只存储在一处**，其他地方用 Markdown 相对路径链接。

## 四、inbox 使用规范

### 原则

- inbox 是**原始资料仓库**，不要求格式整齐，直接粘贴或存入即可
- 存入后不修改原始内容，保留原始来源的完整性

### 文件命名规范

```
YYYY-MM-DD_来源或类型_简短描述.扩展名
```

示例：
```
2026-04-10_周报_多线激光第14周.md
2026-03-01_设计文档_点云匹配方案v3.pdf
2026-02-15_会议记录_跨组接口对齐.md
2025-11-20_设计文档_重定位方案（已过期）.md
```

### inbox 存放位置选择

| 资料类型 | 存放位置 |
|---------|---------|
| 某小组的周报、设计文档、技术讨论 | `teams/<组名>/inbox/` |
| 某项目的需求文档、里程碑、评审记录 | `projects/<项目名>/inbox/` |
| 跨组会议记录、上级要求、全局性文档 | 根目录 `inbox/` |

## 五、信息时效性原则

### 标注规范

结构化文件中每条关键信息须在「来源」列或括号内标注信息来源和日期：

```
| 当前瓶颈 | 雨天点云匹配退化严重 | 2026-04-10 周报 |
```

### 冲突处理规则

- 若两条信息矛盾（如不同时间的文档给出不同指标目标），**不自动取较新的**
- 应暂停填充，列出冲突内容，由 TL 确认后再写入
- 确认后在对应条目备注：`（已与 [来源A] 确认冲突，以 [来源B] 为准，2026-XX-XX）`

## 六、信息填充工作流

```
收集原始资料
    ↓
存入对应 inbox/（按命名规范加日期前缀）
    ↓
提炼结构化信息，填入对应 .md 文件
    ↓
标注来源和日期
    ↓
遇到冲突 → 暂停 → 确认 → 写入
```

## 七、维护节奏建议

| 频率 | 动作 |
|------|------|
| 每周 | 更新 `overview/tasks.md`、`overview/risks.md` |
| 每次 1:1 或例会后 | 更新相关 `teams/*/people.md`，补充 inbox |
| 有新信息时随时 | 丢入对应 inbox，打上日期 |
| 每月 | review `overview/org.md`，检查运行机制是否需要调整 |

## 八、AI 协作规范

以下规则适用于 AI 协助整理材料、填充文档的所有场景。

### 原则一：信息不完整时主动询问

整理材料时若发现提到了某个项目/功能/角色，但材料中没有对应细节：
- 不跳过、不推断
- 主动告知缺少哪类信息，并询问是否能补充
- 等待补充材料到位后再继续

### 原则二：不引入幻觉信息

- 材料中不清楚的内容，直接标注 `【待填充】` 或说明「材料未提及」
- 不基于行业经验、常识或推断自行补充"合理内容"
- 不确定的信息须显式标注：`【待确认】`，并说明不确定的原因

### 原则三：复杂信息先讨论再落笔

当材料信息量大、关联复杂或涉及多个文件更新时：
- 整理前先与用户讨论：说明准备按哪个方向梳理、材料里提到了哪些关键信息
- 等用户核对确认后，再执行写入操作
- 单条信息简单、来源清晰时可直接填入，无需每次都确认

### 总原则

**一切内容必须基于真实材料。材料不到位，宁可留白，不可编造。**

## 九、占位符规范

未填充的内容统一用以下占位符，便于搜索：

- `【待填充】` — 信息尚未收集
- `【待确认】` — 信息已有但需要核实
- `【已过期，待更新】` — 信息存在但可能不准确
