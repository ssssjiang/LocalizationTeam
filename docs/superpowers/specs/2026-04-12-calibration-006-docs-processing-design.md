### D-008：Butchart MCT 产线标定完整工站规格【已定案】

**内容**：来自 `002_Butchart标定与检测：方案设计整理/001_V2`

- MCT 工站：二维码方形场地（3m×3m，挡板高 1m，地面四角贴二维码）+ 滑轨（3m，棋盘格亚光膜，整机改为模组）+ 机械臂（垂直二维码，运动80s采集图像+IMU）
- 场地要求：地面平整硬质（防打滑），挡板稳定不移动，围板贴正
- 图片链接：`calib_butchart_mct_field.jpeg`（来自 P7h4...）、`calib_butchart_traj_1.png`（U8z...）、`calib_butchart_traj_2.png`（WXi...）

### D-009：售后标定轨迹选型决策【已定案】

**内容**：来自 `006_售后标定测试数据采集方案` + `016_售后标定流程文档`

- 验证场景：1.5m×1.5m 墙角，两面墙贴标定板（5列3行）+ 地面二维码
- 轨迹对比：S形（成功率100%但遥控困难，缺少关键帧）；伞形（成功率80%，路线冗余）；工形（成功率60%，路线可优化）
- **Monet 最终轨迹**：逆时针旋转180° → 顺时针135° → 对角线走1.2m → 顺时针90° → 逆时针180° → 顺时针180° 后静止开始计算
- **Versa 最终轨迹**：逆时针45° → 前进0.9m → 掉头前进1.0m → 掉头前进0.1m → 逆时针45° → 逆时针180° → 顺时针135° → 对角线走1.2m → 顺时针90° → 逆时针180° → 顺时针180° 后静止
- 固定轮间距/半径后 yaw 误差 <0.2°，后轮间距最大误差 4.5cm（小场地局限）
- 图片链接：`calib_aftersale_station.jpeg`（GZx...）、`calib_aftersale_traj.png`（whiteboard_1_...201005）

### D-010：Monet 四目+Odo 标定方案设计【已定案】

**内容**：来自 `009_Monet四目+Odo标定方案`

- 方案 A（选定）：类似原左目-Odo 方案，主函数循环3次，分别标定左目/左侧目/右侧目到 Odo；在原框架下实现最快，各相机独立标定无跨机误差
- 代价：耗时约原来3倍，日志量3倍
- 主循环流程：读 json + 图片 → SLAM 建图 → GBA 后定位 → 时间同步 → 标定 → 误差判断 → 写入 json（共3轮）

### D-011：侧目相机外参不需标定，使用结构值【已定案】

**内容**：来自 `017_侧目相机外参标定的意义`

- **结论**：侧目测距精度要求不高，外参标定没有必要，直接使用结构值
- **依据**：① 侧目主要用于活体检测（转向盲区），双目看不到的直行不会压到；② 危险边界依赖前视双目；③ 侧目成像质量差，单目测距精度低，只能辅助识别，不能直接测距沿线；④ 侧目垂直 FOV=118°，安装高度18.01cm，测距盲区10.8cm
- **即使侧目画地图也不需标外参**：障碍物地图更新可依赖前视双目，侧目障碍物地图价值有限
- 合入时间：2026-02-05（`018` bug 列表：关掉侧目标定，toc.z 固定为结构值）

### D-012：产线遮挡检测方案设计【待确认最终选型】

**内容**：来自 `021_产线遮挡检测设计方案`

- 方案一：新建遮挡检测工站（红布前采图 → AT指令生成内参 → 调遮挡检测接口 → 输出json），pros：仅标定人员出代码包；cons：新工站预算 + 联调耗时 + 影响产线效率
- 方案二：在机器上做遮挡检测（给丁毅接口，获取左右目彩图调用接口输出结果），pros：流程简单高效；cons：需和状态机/插件/中间层联调
- 方案三：给装备提供遮挡检测代码自行移植，pros：最简；cons：需装备修改 UI

> **待确认**：当前未明确最终选型，需向邱冰冰确认实际执行方案（标注为 gaps Q-007）

### D-013：外参文件保护 + gdc_mapper 问题清单【已定案】

**内容**：来自 `004_割草机外参文件校验与保护`

- **gdc_mapper 6 个已知问题**：
  1. bin 文件存 `/mnt/reserve/gdc/`，无法随模组携带
  2. EEPROM 4KB 不够存 bin 文件（13KB）
  3. bin 文件存在损坏可能
  4. bin 文件本质是内参的冗余存储（数据存两份）
  5. gdc_mapper 会替换 `dual_calibration.json`（直接导致 Monet MK2 标定结果全部错误）
  6. 生成 bin 耗时约数十秒（主要在写 txt 步骤，3276ms）
- **解决方向**：外参文件保存 EEPROM + 中间层消息发出；gdc_mapper 提供在线生成 remap 表接口

### D-014：重投影误差检测算法 v2【已定案】

**内容**：来自 `010_二驱标定算法检测重投影误差检测`

- v1（世界坐标系）：Pw2 = Tlf⁻¹ × Tcf × (Tco × Tdelta_o × (Toc × Pw1))，误差波动 0.3–7.5px，受 camera pose 影响大
- **v2（相机坐标系，选定）**：首帧相机坐标系下3D点投影到当前帧，误差波动收窄至 0.3–3.9px（正常数据集），大部分 ≤1px
- 打滑检测阈值：平均重投影误差 >2px 或标准差 >2px → 提示轮子可能打滑/卡死（依据：打滑数据集平均误差 6.18px，正常 ≤1.65px）
- 图片链接：`calib_reproj_algo_v2.png`（Toxy...）

### D-015：温度补偿对焦距影响量化【参考】

**内容**：来自 `003_温度补偿对焦距影响的仿真`

- 参考扫地机系数：kx = 0.07~0.0725（随温度区间变化），deltaT = currT - calibT
- 极限工况：工厂20°C + 工作70°C（deltaT=30°）→ fx/fy 最大变化 +2.25px；工厂30°C + 工作0°C（deltaT=-30°）→ 最大变化 -2.175px
- 公式：fx_new = fx_fac + kx × deltaT
- 用途：判断是否需要引入温度补偿模块的量化依据

## 三、problems.md 新增条目（P-004 ～ P-006）

### P-004：行差超标问题（B1-37 机型）

- **症状**：B1-37 部分图像对行差 >1.22px，不通过行差检查
- **排查**：`slam_data_toolkit` 测试 105 lake 数据，648/1264 对图像超标，图像左上/中上行差为正、右上为负，确认为 FOV 差异导致的局部行差不均匀
- **根因**：双目模组 FOV 差异 + 校准参数偏差叠加
- **修复**：行差检测及行差标定功能，2026-03-02 合入 prebuild/develop
- **检测工具**：`slam_data_toolkit` 双目相机分析工具，SIFT 特征点 + 光流，支持静态/动态场景，生成行差分布直方图
- **来源**：`007/006_行差问题复现与检测方案`、`007/007_B1-37行差&Cam Time问题排查`、`018`

### P-005：gdc_mapper 替换 dual_calibration.json 导致 Monet MK2 标定结果全部错误

- **症状**：Monet MK2 批次标定结果全部异常
- **根因**：gdc_mapper 在生成 bin 文件流程中意外替换了 `dual_calibration.json`（问题5，见 D-013）
- **影响**：批量机器标定失效，需重新标定
- **修复方向**：限制 gdc_mapper 写入范围，不覆盖标定结果文件
- **来源**：`004_割草机外参文件校验与保护`

### P-006：售后标定纸板治具平放翘边

- **症状**：纸板治具（厚4.3cm，65cm×150cm）放平于地面后有翘边，影响二维码平面度
- **验证数据**：4次标定结果 yaw 极差约1°（与产线 MCT 相比），纸板表面有肉眼可见凹凸纹路
- **影响**：标定结果 yaw 稳定性差（极差1°），roll/pitch 正常（极差<0.3°）
- **改进方向**：换硬质治具材料，保证平面度
- **来源**：`022_售后标定治具验证`
