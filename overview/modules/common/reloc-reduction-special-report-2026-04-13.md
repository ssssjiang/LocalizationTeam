## 1. 结论摘要

当前激光和融合两组都在沿着同一条主线推进：**把“暂停/恢复/轻微搬动后立即重定位”的保守策略，逐步收缩成“只有明确搬动或定位质量持续变差时才触发重定位”**。

这条主线不是简单地“取消重定位”，而是把原本偏一刀切的触发逻辑，改造成更细粒度的分层策略：

- 能靠轮速或递推 hold 住的场景，优先不重定位
- 轻微搬动、刹车、短时断流等场景，优先局部补偿或继续计算
- 明确搬动、局部重定位失败、连续多帧匹配差等场景，再升级到局部或全局重定位
- 在“导航是否静止”“是否检测到搬动”之外，再增加“恢复后的实际匹配质量”作为兜底判断

从现有材料看，**激光侧的重点是减少暂停恢复和断流场景下的重定位触发，融合侧的重点是减少轻微搬动和 pause 期间状态切换带来的重定位频率。**

## 2. 起点设计：为什么最初策略偏保守

全组最早的 `SLAM暂停/恢复` 设计里，pause/resume 的核心目标是省电、省 CPU、省内存，因此暂停期间采用了非常保守的策略：

- pause 后持续给导航发送暂停前的 pose
- pause 期间不 dispatch 传感器
- pause 期间不检测传感器，不进行重定位
- 但如果 pause 期间发生搬动、gyro 超阈值、odom 移动，则由导航记录状态，在 resume 时触发重定位

这套基线说明，系统一开始更重视的是**安全恢复**和**状态确定性**，而不是减少重定位次数。

其内在假设是：

- 暂停期间 SLAM 自己对外界变化感知有限
- 恢复后若机器姿态或位置已变，继续沿用 pause 前状态风险较高
- 因此宁可在恢复时走重定位，也不要直接相信 pause 前 pose

这并不等于“所有 pause 后都重定位”，但它确实是一种**偏保守、偏导航主导**的恢复设计。

## 3. 激光侧：从暂停恢复重定位，演进到按场景减触发

### 3.1 暂停恢复文档已经体现出“不要一刀切重定位”

`暂停恢复-定位优化.md` 已经把 pause/resume 拆成多类情况，而不是统一处理：

- 无搬动、无移动：轮速递推
- 小位移推动：导航发起局部重定位；失败再升级到全局重定位
- 明确搬动：当前由导航直接发起全局重定位
- 断流导致的 pause/resume，若中间未完全停止：中间不发局部重定位，先靠轮速递推兜底
- 只有轮速兜不住时，后续才考虑由 SLAM 自主发起局部或全局重定位

这说明激光侧在暂停恢复问题上的核心变化，不是“新增一个主动重定位”，而是先把大量本来可能直接走重定位的情况，改成**先轮速兜底、先继续递推、先不打断流程**。

### 3.2 D-016：恢复后按“匹配质量”再决定是否升级

`D-016 暂停/恢复定位优化方案` 把激光侧的当前思路进一步明确为两层：

第一层是场景分流：

- 无搬动：轮速递推
- 小位移：局部重定位
- 搬动：全局重定位
- 断流且未完全停止：先不发局部重定位

第二层是质量兜底：

- 若暂停恢复后连续多帧匹配率和得分都低于阈值，则由定位主动通知导航发起全局重定位

这条“主动全局重定位”看起来比原始设计更激进，但它的真实作用不是扩大重定位范围，而是把触发条件从“只依赖是否搬动”扩展成“还看恢复后的定位质量”。  
换句话说，它允许系统在很多场景里**先不重定位**，只有实际 hold 不住时再升级。

### 3.3 D-022：减少对“导航必须完全静止”的依赖

`D-022 局部重定位减少对导航静止依赖优化` 是激光侧“减少重定位”更明确的一步。其背景是：

- 现有 setpose 依赖导航等待机器完全静止
- 这会带来等待延迟
- setpose 计算期间主线程可能阻塞或漂移

文档提出三阶段方案：

- 短期：setpose 前后加轮速补偿，覆盖静止等待期的小运动
- 中期：局部重定位异步计算，主线程继续 SLAM 递推和发布
- 长期：导航先走，局部重定位后台算；失败时再主动发起全局重定位

这条线的本质不是把重定位做得更频繁，而是把局部重定位从“必须等、必须停、容易扩大成全局流程”，变成更轻量、更延后决策的机制。

### 3.4 激光侧已出现明确的“取消/减少重定位代码合入”

`mslam/timeline.md` 记录了一个非常直接的节点：

- `2026-03-25`：取消/减少重定位代码合入

同时 `P-011` 又记录了该变更合入时曾触发 fusion coredump。  
这说明“减少重定位”在激光侧不是口头目标，而是已经落到了实际代码路径和跨模块接口调整上。

## 4. 融合侧：目标更明确，直接写成“轻微搬动不重定位”

### 4.1 融合侧需求直接提出“减少重定位”

`RTK割草机搬动.md` 开头就写得很直接：

- 减少重定位，轻微搬动情况下不进行重定位
- 搬动中仍希望显示位置

文档的核心判断是：真实使用中，机器经常是“先 pause，再搬动，再恢复”，如果 pause/resume 与搬动处理绑定得过于机械，就会频繁触发重定位。

因此文档开始讨论两类策略：

- 用 pause 作为切换模式的入口，在暂停期间覆盖大部分抬起/搬动场景
- 或新增独立的抬起/放下状态，而不是把所有情况都等价成重定位

这说明融合侧已经不再把“检测到 pause + 运动”自动等同于“必须重定位”，而是在主动寻找**更少打断流程的状态切换方式**。

### 4.2 融合版本记录中，已经出现“减少重定位频率”的实现

`fusion/versions.md` 有一个非常关键的版本记录：

- `2026-03-02`：暂停期间保持计算，RTK 位置和 imu 角度分离，减少重定位频率

这条记录很重要，因为它表明融合侧并不是只在讨论“搬动后怎么判”，而是在更底层地改变 pause 期间的状态维护策略：

- 不再简单冻结或等待恢复后整体重置
- 而是在 pause 期间继续保留一定计算能力
- 用更细的状态管理，减少恢复后不得不重定位的次数

### 4.3 融合时间线也显示：重定位状态机曾因 pause/搬动互相打断

融合时间线里的几个节点可以串成一条线：

- `2025-09-25`：SLAM pause/resume 功能合入
- `2025-11-13`：pause 打断重定位状态 bug 修复合入
- `2026-01-22`：搬动打断主动重定位 bug 修复合入

这说明融合侧的问题不只是“重定位太多”，还包括：

- pause 状态
- 搬动重定位状态
- 主动重定位状态

这几条状态流曾经互相打断、互相覆盖。  
因此融合侧“减少重定位”的真实目标，也包含**减少状态机相互抢占**，让 pause、搬动、主动重定位之间的关系更稳定。

## 5. 激光与融合的共性：都在把“是否重定位”变成分层决策

从两边材料汇总后，可以看到一个共同趋势：


| 层级            | 激光侧                             | 融合侧                             |
| ------------- | ------------------------------- | ------------------------------- |
| 第一层：能不能先不重定位  | 轮速递推、断流期间先兜底、不完全停止时不立刻发局部重定位    | pause 期间继续计算、轻微搬动不重定位           |
| 第二层：能不能只做局部修正 | 小位移时优先局部重定位；setpose 异步化减少等待和阻塞  | 通过状态切换覆盖 pause+搬动的大多数场景，而非一律重定位 |
| 第三层：什么时候升级    | 局部重定位失败，或连续多帧匹配率+得分差时升级到全局/主动全局 | 明确搬动、遮挡导致丢定位、主动重定位状态保持失败时再升级    |


因此，“减少重定位”并不是一个单点功能，而是一组共同原则：

- 优先保持当前状态连续性
- 优先让轻微扰动被补偿掉
- 优先让局部问题在局部解决
- 只有在证据充分时才升级到全局重定位

## 6. 当前仍未完全统一的问题

虽然激光和融合都在朝减少重定位推进，但目前材料仍显示两边还有几类未完全统一的问题。

### 6.1 搬动阈值和正常刹车的边界

激光侧区分了“无搬动”“小位移”“搬动”，融合侧则明确提出“轻微搬动不进行重定位”。  
但仓库里的本地材料还没有给出一套全局统一、跨模块共用的阈值定义。

仍待统一的问题包括：

- 小位移的距离 / 角度阈值是否一致
- pause 后正常刹车是否会被误判为搬动
- 轻微搬动与真正抱起、抬起之间如何区分

### 6.2 导航事件与定位质量，谁优先

目前至少有两类触发源：

- 导航/状态机检测到 pause 期间搬动
- 定位模块在恢复后检测到连续多帧匹配差

如果两者同时存在，或者结论相反：

- 导航未判搬动，但定位持续匹配差
- 导航判为搬动，但定位短时间内仍能 hold

目前本地材料中还没有看到一份统一说明，明确这两类信号的优先级和合并规则。

### 6.3 “主动重定位”仍然带有安全边界

激光 `D-022` 的长期方案里已经明确写出：  
若导航已先开始走，后台局部重定位失败后再主动发起全局重定位，存在安全风险。

这意味着“减少重定位”和“晚一点再触发重定位”虽然提升了流畅性，但也把系统推向了另一个平衡点：

- 太早重定位：恢复慢、频率高、体验差
- 太晚重定位：导航可能已在错误 pose 上继续行走

当前两组都在寻找这个平衡点，但尚未看到完全收敛后的统一规则。

## 7. 建议的统一理解框架

为了后续继续整理或对外汇报，可以把“减少重定位”统一表述成下面这套框架：

> 减少重定位不是减少安全兜底，而是把重定位从默认动作降级为升级动作。系统优先依赖轮速递推、pause 期间继续计算、轻微搬动补偿、局部重定位异步化等手段维持状态连续性；只有在明确搬动、局部重定位失败或恢复后连续多帧匹配质量恶化时，才升级为全局或主动全局重定位。

这套表述同时覆盖了：

- 最早的 pause/resume 保守设计
- 激光侧的 D-016 / D-022
- 激光侧“取消/减少重定位代码合入”
- 融合侧“轻微搬动不重定位”
- 融合侧“暂停期间保持计算，减少重定位频率”
