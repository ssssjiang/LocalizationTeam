## 一、演进脉络：断流 1.0 → 减少重定位次数 → 断流 2.0

雷达断流相关方案不是一次到位的，而是**沿"有断流就停机重定位 → 尽量减少不必要的重定位 → 断流期间无感恢复"三步递进**。三个阶段同样重要，**任何一个被回滚或绕过都会让上一代问题回潮**。


| 阶段                                   | 触发动机                                           | 核心策略                                             | 上线节奏                                | 对应决策                  |
| ------------------------------------ | ---------------------------------------------- | ------------------------------------------------ | ----------------------------------- | --------------------- |
| **1.0 断流 1.0**                       | 支持「断流后机器不死机」的最低需求                              | 断流 → **机器停机** → 触发重定位（轮速递推 + 导航 set pose）        | 历史方案，已替代                            | D-007                 |
| **1.1 减少重定位次数优化**（独立阶段，**不是为了断流**而做） | 导航因断流 / 搬起 / lidar 关闭等场景**频繁外发重定位指令**，体感差、误触发多 | 收到重定位指令后算法**先尝试自恢复定位**，**成功则反馈取消字段**让导航撤回（versa） | **HF1.1 已合入**                       | D-016 / D-022 / D-025 |
| **1.2 断流 2.0**                       | 在 1.0 基础上追求「**断流期间无感恢复**」                      | 断流后 **6s 内减速运动 + 局部重定位无感恢复**，仅在失败时才走重定位流程        | **计划 HF2.2 合入，目前仍在联调**（4/21 起王亚萌主推） | D-008                 |


> **核心思路**：从"有断流就停机重定位"演进到"尽量减少不必要的重定位，只在真正需要时才触发"。  
> 减少重定位次数优化是 1.0 与 2.0 之间的**独立优化轴**，触发动机是「重定位频率本身就是体验问题」，与断流方案正交；2.0 上线后该机制依然保留作为兜底。

### 1.1 阶段一：断流 1.0（D-007，历史方案，已被 D-008 替代）

**背景**：lidar 时间同步断流通常出现在 2s 量级，500ms 起报。最初目标只是"机器在断流后不要死机"。

**设计要点**：

1. **导航侧记录 + 轮速递推**：导航在收到断流（暂停）信号时记录最近一帧 SLAM pose，断流期间用轮速 diff 递推一个临时 pose 给到融合，避免融合断帧。
2. **lidar 恢复后导航 set pose**：lidar 数据回来时，导航把递推后的 pose 通过 `set_pose` 接口发给 SLAM 做匹配初值，由 SLAM 完成局部重定位。
3. **断流报告阈值 = 500ms**：低于 500ms 不报断流（仅靠 odom 递推兜底）；超过则触发上述链路。

**已知缺陷（推动 1.1 / 2.0 的直接原因）**：

- **强依赖 set pose**：断流恢复必须等导航发指令，导航逻辑一改就要联调。
- **断流即停机即重定位**：哪怕只是几百 ms 的小毛刺，也会触发一次完整重定位 → 导致**用户体感「机器一直在转圈/停顿」**。
- **斜坡场景兜底差**：500ms 的兜底窗口太短，一旦轮速积分有误差，就会重定位失败。
- 2026-01 底因 SDK RTM 分支加打印日志、断流频率猛增（几分钟内十余次甚至数十次），1.0 完全顶不住，**直接催生了 2.0 立项**（2026-02-05 王亚萌加入断流专项）。

**仓库映射**：

- 设计原文：`teams/laser/inbox/003_多线激光/006_算法文档/015_雷达断流/001_雷达机器断流后slam动作/雷达机器断流后slam动作.md`
- 决策记录：`teams/laser/modules/mslam/decisions.md` D-007（已标注被 D-008 替代）
- 时间线：`teams/laser/modules/mslam/timeline.md` 2026-01 底「雷达断流问题高频爆发」、2026-02-05「王亚萌加入断流专项」

### 1.2 阶段二：减少重定位次数优化（versa，HF1.1 已合入）

> **设计文档**：`teams/laser/inbox/003_多线激光/006_算法文档/015_雷达断流/003_versa重定位优化/versa重定位优化.md`（2026-04-22 归档）  
> **闫冬流程图 & 测试结果**：飞书 wiki [取消重定位算法流程及测试结果](https://roborock.feishu.cn/wiki/FQ2hwK73YizmxIkNIepc802DnNd)  
> **前提**：搬起对 lidar 无感；**关闭 lidar 影响较严重**——这条路径主要解决的就是 lidar 关闭/断流诱发的重定位指令风暴。

#### 1.2.1 设计思想

1. **导航维持原来的发起逻辑不变**（不动导航主流程，降低引入风险）。
2. **抬高局部重定位的发起阈值**（导航需要：**1 弧度 / 70 cm** 内发起局部重定位；阈值升级后改成单指令）。
3. **在 acc（全局重定位 accumulate）开始时增加「定位扛得住就反馈取消」的逻辑** —— 这是减少次数的最关键开关，让 SLAM 在重定位真正开跑前有机会"踩刹车"。

#### 1.2.2 定位模式流程


| 触发信号                            | SLAM 动作                                          |
| ------------------------------- | ------------------------------------------------ |
| **暂停信号**                        | 不管暂停与否，只要有 lidar，内部都计算一遍；这样**搬动与否对内部一致**，不引入额外分支 |
| **恢复 / 直接收到全局重定位 acc**（无暂停、搬起等） | 走下面分支判断                                          |
| → 导航发起**局部重定位**                 | 沿用原 SLAM 局部搜索逻辑（导航侧合成 1rad/70cm 单指令）             |
| → 导航发起**全局重定位**                 | SLAM 评估当前匹配状态（已考虑 lidar 关闭等），通过**取消重定位字段**与导航交互  |
| → 匹配 OK                         | 在 acc 阶段返回**取消重定位字段（"扛得住"）**，**不用转圈，机器继续走**      |
| → 匹配不行                          | **继续走完重定位流程**                                    |
| 兜底                              | 若导航没发重定位但定位自感不行，可**主动重定位**                       |


> **接口实现**：当前**复用重定位 prepare 字段**——若 prepare 返回 false，导航撤销本次重定位；否则继续重定位。后续可演进为独立的取消字段。

#### 1.2.3 建图模式流程（lidar 统一不关闭，只有断流有影响）

1. **新增局部重定位**：导航在断流恢复后下发局部重定位，**单指令 set pose** 即可（与定位侧对齐，把现在的「切模式 + 加载地图 + set pose」三指令合并为单指令）→ **扩展建图与建图模式都覆盖**。
2. **搬起场景**：搬起会让已有建图直接失败，无需 SLAM 侧额外处理。

#### 1.2.4 主动重定位接口（定位模式预留）

设计阶段先预留好接口，目前定位侧无主动触发逻辑，**留给后续 2.0 / 主动重定位扩展**。

#### 1.2.5 对导航的潜在改动

属于"体验侧润色"，让一闪而过的状态不要骚扰用户，由导航自评估：

- 局部重定位时 **app 不提醒**（但允许遥控）。
- 全局重定位 **acc 没反馈之前 app 不进入"重定位状态"**，避免导航发了又取消造成 UI 抖动。

#### 1.2.6 收益

1. **平滑演进、引入问题少、可控**（导航主流程不动）。
2. **最大限度减少重定位次数**——能扛住就不重定位。
3. **能兜底部分逻辑遗漏**（acc 反馈是 SLAM 自己做的最后一道闸）。
4. **后续优化空间更大**：进一步压缩重定位只需 SLAM 自己升级算法，无需再次拉导航联调。

#### 1.2.7 仓库映射

- 设计原文：`teams/laser/inbox/003_多线激光/006_算法文档/015_雷达断流/003_versa重定位优化/versa重定位优化.md`（含两张设计图：`whiteboard_1_1776825233365.png` / `Z5dUba1fGotF4exz8SMcpxY3nPg.jpeg`）
- 决策映射：`decisions.md` D-016（暂停时优化）/ D-022（短期防护 + 主动重定位机制）/ D-025（搬起放下减少重定位）
- 合入版本：**versa HF1.1**

### 1.3 阶段三：断流 2.0（D-008，计划 HF2.2，仍在联调）

> **设计原文**：`teams/laser/inbox/003_多线激光/006_算法文档/015_雷达断流/002_断流2.0/断流2.0.md`

#### 1.3.1 设计目标

1. **不停机；6s 才停机**（把停机时机大幅推后，争取 6s 窗口让 SLAM 自己救回来）。
2. **导航尽量无感**（不再依赖 set pose；SLAM 内部自治）。

#### 1.3.2 风险与挑战

- 断流持续时间未知，仍需阈值兜底防止无限滑行。
- 建图会走到未知区域，与定位逻辑不同，必须**分支设计**。
- 定位不准时的**安规问题**（漏割草等）由产品侧承接，本方案不覆盖。

#### 1.3.3 分场景策略


| 场景                      | 断流时                                                 | 恢复后                                                                                    |
| ----------------------- | --------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **定位 · 6s 以上**          | 中间层报错状态 → **直接停机**                                  | 等待 lidar 恢复，**导航无需处理可立即移动**；定位移动后内部兜底计算，扛不住才发起主动重定位                                    |
| **定位 · 6s 以内**（断流 1s 起） | 中间层把"断流开始"信号给导航 → **减速 0.2 m/s / 0.3 rad/s**，导航确认即可 | SLAM 内部：**轮速先补偿 → 匹配 → 失败则启动局部重定位（结果用轮速补偿到当前激光位置再匹配）**。任一成功则其他模块无感恢复；都失败则发起**全局重定位请求** |
| **建图 · 1s 以上**          | **直接停机**（保成功率，复用现有逻辑改阈值，姚远侧改动到 1s）                  | 内部兼容设计参考定位侧；都失败则向导航发**建图失败反馈**（第一阶段先上报，后续再优化）                                          |


#### 1.3.4 关键兼容设计（ps 章节）

1. 断流期间**融合模块正常发布位姿**。
2. 断流期间**多线 SLAM 正常 odom 递推**，但**不修正、不发布**。
3. 恢复时**多线使用自己递推的 pose 做局部重定位的初始位姿**，**不再使用导航的 set pose**。
4. **动态做局部重定位**（不强求机器停下）。
5. 成功后给融合 set pose 继续工作；失败 → 主动全局重定位。
6. 断流期间计时，**超 6s 直接停机报错**。
7. 局部重定位计算时**若再次断流（套娃）→ 直接返回失败**，计时与上次累加。
8. 断流恢复后用**轮速前后递推值**做局部重定位 + **轮速并行递推**压低时间；成功后再补一次匹配；失败则直接全局重定位（救回意义不大）。

#### 1.3.5 已知风险

- **斜坡补偿缺陷**：兜底窗口从 500 ms 拉到 6 s，斜坡场景下补偿算法精度不足，断流时定位失败概率上升。优化方向：高度积分 + 重力对齐角度积分（→ P-007）。

#### 1.3.6 仓库映射

- 设计原文：`teams/laser/inbox/003_多线激光/006_算法文档/015_雷达断流/002_断流2.0/断流2.0.md`
- 决策记录：`decisions.md` D-008（含建图/定位分支与 ps 兼容章节）+ D-015（轮速引入断流补偿）
- 关联问题：`problems.md` P-006（恢复失败场景）/ P-007（斜坡补偿缺陷）
- 联调状态：见 W16 周报「主线 1：断流 2.0」段落，4/21 起王亚萌负责 rebase + drop 阈值最终敲定

## 四、雷达断流原因分析


| 原因           | 说明                                                            |
| ------------ | ------------------------------------------------------------- |
| **GPS 报文阻塞** | 模拟 GPS 报文因 AP 实时性出现阻塞，最大可能有几帧阻塞                               |
| **时间戳回退**    | 雷达收到延迟的 GPS 报文，导致时间戳回退引发断流                                    |
| **硬同步失效**    | 时间同步问题，模拟信号无法满足硬同步要求                                          |
| **厂家强制关闭**   | **大疆厂家会强制关闭 2s** 也会造成断流（来自本次工作汇总，非现有 D-008 文档记录）              |
| **SDK 加印日志** | SDK RTM 分支加打印日志影响实时性，是 2026-01 底断流频率显著上升的诱因之一（见 D-008 历史背景章节） |


> ⚠️ 「大疆厂家强制关闭 2s」此前未在 `decisions.md D-008` 的根因章节出现，本次专题汇总首次记录；建议向飞书群讨论 / SDK 文档复核确认。
