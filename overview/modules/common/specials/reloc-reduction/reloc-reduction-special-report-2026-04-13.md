## 1. 起点：保守设计的假设

最早的 `SLAM暂停/恢复` 设计以**省电、省 CPU、安全恢复**为目标，pause 期间策略极保守：

- pause 后持续向导航发送 pause 前的 pose，SLAM 停止分发传感器
- pause 期间不检测状态，不触发重定位
- 但若期间发生搬动（gyro 超阈值 / odom 移动），则导航在 resume 时无条件触发重定位

**这套设计的内在假设**：暂停期间 SLAM 对外界变化感知有限；恢复后若姿态已变，直接沿用旧 pose 风险高；宁可多走一次重定位，也不赌状态连续。

这并不是错误设计，而是在安全性和流畅性之间的早期权衡——但随着场景细化，它带来了大量不必要的重定位触发。

---

## 2. 核心原则：把重定位从默认动作降级为升级动作

激光和融合当前共同遵循的分层原则：


| 层级             | 判断依据                  | 策略                 |
| -------------- | --------------------- | ------------------ |
| 第一层：能不能不重定位    | 断流短 / 无移动 / 轮速能兜住     | 轮速递推或继续计算，不打断流程    |
| 第二层：能不能只做局部修正  | 小位移 / set_pose 可以解决   | 局部重定位（尽量异步，不阻塞主线程） |
| 第三层：什么时候必须全局重定 | 局部失败 / 连续多帧匹配差 / 明确搬动 | 升级到全局重定位 / 主动全局重定位 |


> **一句话概括**：优先保持状态连续性，让轻微扰动被补偿掉，让局部问题在局部解决，只有证据充分时才升级到全局重定位。

## 3. 激光侧机制

### 3.1 场景分流与质量兜底（D-016）

**设计背景**：原始策略对 pause/resume 一刀切，不区分是否真的移动了。D-016 把场景显式拆开：


| 场景                | 策略                              | 实现状态  |
| ----------------- | ------------------------------- | ----- |
| 无搬动、无移动           | 轮速递推，不重定位                       | ✅ 已实现 |
| 小位移推动             | 导航发起局部重定位；失败升级全局重定位             | ✅ 已实现 |
| 明确搬动              | 导航直接发起全局重定位                     | ✅ 已实现 |
| 断流且未完全停止          | 先不发局部重定位，轮速递推兜底                 | ✅ 已实现 |
| 恢复后连续多帧匹配率 + 得分均差 | SLAM 主动上报 `relocate_flag`，导航发全局 | ✅ 已实现 |


> **设计意图**：最后一条"质量兜底"是关键扩展——它允许前几层"先不重定位"，只有实际 hold 不住时才由定位侧主动升级，而不是保守地预防性重定位。

> ❓ 主动重定位的匹配率 + 得分阈值待联调确认（见 Q-2）。

**代码节点**：取消/减少重定位代码合入 `2026-03-25`（`mslam/timeline.md`）；合入时曾触发 fusion coredump（P-011），已修复。

---

### 3.2 局部重定位去阻塞（D-022）

**设计背景**：现有 `set_pose` 要求导航等机器完全静止后才调用，且调用期间主线程阻塞。这带来两个代价：等待延迟 + 阻塞期间 pose 漂移。D-022 把 `set_pose` 从同步阻塞改造成渐进异步。

#### ✅ 短期方案（已实现）

`set_pose` 前后加轮速补偿，覆盖静止等待期的少量运动。框架不变，影响范围最小。

#### ✅ 中期方案（已完成，2026-04）

`set_pose` 触发后开启独立线程执行局部重定位；主线程继续 SLAM 递推和发布（不刷新地图）。完成后：成功 → 修正初始 pose + 刷新地图；失败 → 返回 false，导航发起全局重定位。

**效果**：消除了主线程阻塞，"必须等停"的依赖显著降低。

#### 🔴 长期方案（暂缓 — 安全风险未解决）

`set_pose` 立即返回 true，导航不等待直接开始行走；局部重定位后台算，失败时再主动发起全局重定位。

> ⚠️ **风险**：导航已开始行走，但 pose 尚未修正，定位跑偏期间存在安全隐患。当前暂缓，待安全机制补充后推进。

---

### 3.3 断流 2.0：把停机时机推后（`断流2.0.md`）

**设计背景**：原有策略断流即停机，停机即触发重定位。断流 2.0 的目标是**不停机（6s 才停机）**，把原本"断流→停机→重定位"这条默认路径，在重定位之前拦截。

**定位模式下的三级兜底策略**：


| 级别   | 触发条件      | 策略                                   | 实现状态  |
| ---- | --------- | ------------------------------------ | ----- |
| 一级兜底 | 断流 > 1.0s | 导航减速降转速（0.2m/s / 0.3rad/s）           | ✅ 已实现 |
| 二级兜底 | 持续断流      | SLAM 轮速补偿 + 匹配尝试                     | ✅ 已实现 |
| 三级兜底 | 二级补偿失败    | 局部重定位（用轮速递推 pose 作为初始值，不等导航 setpose） | ✅ 已实现 |
| 升级全局 | 局部重定位失败   | 向导航发起全局重定位请求                         | ✅ 已实现 |
| 停机   | 断流超过 6s   | 停机                                   | ✅ 已实现 |


**建图模式**：断流 1.0s 直接停机（保成功率），恢复后参考定位侧兜底设计。

> **与 D-022 的关系**：D-022 解决"局部重定位如何不阻塞主线程"；断流 2.0 解决"断流恢复时不再等导航 setpose，改为 SLAM 自身轮速递推初始 pose"。两者配合，整个断流恢复路径对导航的依赖显著降低。

> ⚠️ **待跟进**：严重退化时兜底当作断流处理（兼容现有策略）为新增扩展需求，尚在排期中（王亚萌）。

## 4. 融合侧机制

### 4.1 搬动不重定位（`RTK割草机搬动.md`）

**设计背景**：真实使用中，机器经常"先 pause，再搬动，再恢复"，如果 pause + 运动机械绑定为重定位，频率会很高。

**当前方向**：

- 用 pause 作为模式切换入口，在暂停期间覆盖大部分抬起/搬动场景
- 引入独立的抬起/放下状态，而不是把所有情况等价成重定位
- 核心判断：**轻微搬动不触发重定位**


| 场景      | 策略            | 实现状态  |
| ------- | ------------- | ----- |
| 轻微搬动    | 不触发重定位，状态切换维持 | ✅ 已实现 |
| 搬动中显示位置 | 融合模块继续发布位置    | ✅ 已实现 |
| 明确抬起放下  | 融合触发重定位       | ✅ 已实现 |


> ❓ "轻微搬动"与"明确抬起"之间的具体判据（距离/角度/传感器信号组合）尚无跨模块统一定义（见 Q-1）。

---

### 4.2 暂停期间保持计算（`fusion/versions.md`）

**版本节点**：`2026-03-02` — 暂停期间 RTK 位置和 IMU 姿态分离，保持一定计算能力，减少重定位频率。

**设计意图**：不再简单冻结或等待恢复后整体重置，而是在 pause 期间维护更细的状态，减少恢复后"不得不重定位"的次数。


| 场景         | 旧策略      | 新策略                 | 实现状态  |
| ---------- | -------- | ------------------- | ----- |
| pause 期间   | 冻结所有状态   | RTK 位置 / IMU 角度分离维护 | ✅ 已实现 |
| resume 后首帧 | 大概率触发重定位 | 状态连续，按质量决定是否重定位     | ✅ 已实现 |


**状态机稳定历史**（说明这条路不是一帆风顺）：

- `2025-09-25`：SLAM pause/resume 功能合入
- `2025-11-13`：pause 打断重定位状态 bug 修复
- `2026-01-22`：搬动打断主动重定位 bug 修复

> pause、搬动重定位、主动重定位三条状态流曾互相抢占。"减少重定位"的融合侧目标，也包含**减少状态机相互覆盖**。
