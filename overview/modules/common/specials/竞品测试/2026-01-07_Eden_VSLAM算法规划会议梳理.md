## 一、Terramow 竞品摸底测试结论

### 1.1 建图行为

| 维度 | 结论 |
|---|---|
| 建图方式 | 弓字 +（最后）自主沿边 2 圈；弓字宽度 = 机身 |
| 建图模式 | 直接建图（弓字 + 避障）/ 割草建图（更密的弓字，边割边建） |
| 避障策略 | 建图时遇障即避；正常割草不严格避障，会压线材 |
| 建图后处理时间 | 地图保存约 10–20s（疑似落盘，非优化） |
| 回桩后是否再优化 | 无明显优化耗时，建完图 5s 即可开始割草 |
| 地图实时性 | **建图和割草过程都实时更新地图**；显示层在运动中做平滑收缩，静止时显示更多细节 |

### 1.2 建图中异常处理

- **支持建图中搬动机器**：搬动后 APP 在已有半图上重定位，定位中不显示姿态，恢复后正常
- 搬动到 1m 外 b 点：先随机角度转圈尝试重定位，成功后从当前点继续工作（**不回原始 a 点**）
- **打滑几乎不影响定位**，APP 轨迹无错乱
- **遮挡触发重定位**：直接进入「定位中」，疑似基于 VIO 跟踪点数判断；遮挡后只能重定位到老图（不会切到新全局图）
- **不支持首建多区域**，需结束本次再添加；后续可任意增删区域

### 1.3 定位精度与异常工况

- **照度策略**：严格按时间（日出/日落，含位置经纬度计算），与亮度无关；落山时刻自动停止
- **中途搬动充电桩（连桩带机器）→ 出桩位置错，机器一路错下去，不会主动重定位**
  - 仅遮挡 / 跟丢 / 主动触发，才会重定位
  - 偶发：长时间割草后曾自恢复一次，未完全确认
- **重定位成功率非常高，无假成功**；充电桩搬动不影响重定位结果（仍用老图桩位）
- 动态障碍物有一定处理能力（APP 位置正常）
- 创建通道后每次轨迹略有偏动

### 1.4 重定位异常分支（关键发现）

| 场景 | 行为 |
|---|---|
| 建图**已完成** + 搬到无共视区域 | **持续盲走 ≥5min 不报错**，直到偶遇能定位的位置；不符合安规 |
| 建图**已完成** + 搬到区域一旁边（有共视） | 重定位成功并报「界外」 |
| 建图**未完成** + 搬到无共视的 b 区 | 一直尝试重定位；若搬回 a 区可成功；若 b 区能定位，会扩建新区域（待确认） |
| a 区删除后搬到 a 区 | 仍能用老图重定位成功；APP 显示在「空地图」上行走 |
| a/b 区 + 通道，删除 b 区 | b 区地图重新建出并跨区域连通；通道似仍保留（待验证） |

## 二、李宝玉的算法分析总结

### 2.1 Terramow 关键技术推断

- **CPU**：RK3588（Eden 因黑名单确定继续用 X5；XX7 太贵预研中；全志 736 受制裁风险）
- **相机**：SC132DS（与 Eden 相同型号），但**仅做 IR 灰度**用于 VSLAM；Eden 是 RGBD 绿光片，需兼顾感知与 SLAM
- **IMU**：李圳推测为 TDK（与石头 360 同家），普通档；不一定走 7081 高阶料
- **抗打滑能力强、不强依赖 ODOM**：推测靠 VIO + 高频局部图回环，不强依赖里程计
- **多区域 / 多通道建图体验好的根因**：**地图实时优化**，无显式优化耗时

### 2.2 OKVIS 与 Maplab 两套方案对比

| 维度 | OKVIS（Fullgraph 版本） | Maplab |
|---|---|---|
| 框架特点 | 前后端分离：前端 local BA + 后端 Fullgraph 全局 BA + finalba | 全局 BA，依赖完整离线优化 |
| 精度 | 较低（误差均摊 + 部分 BA） | 较高（全局 BA） |
| 建图中重定位 | **天然支持**（multi-session，加载 fullgraph 即可） | 不支持，需开发 |
| 建图中精度 | 有回环 + 误差均摊，会有姿态突变（需通知导航） | 仅靠 VIO，精度差 |
| 重定位后续接 | 需开发 | 需开发 |
| 优化时间 | local 实时；fullgraph 后端可后台跑 | 当前 5min（目标 2–3min） |
| 多帧重定位校验 | 需开发 | 需开发 |

### 2.3 OKVIS Fullgraph 内部机制（讨论澄清）

- **real time graph**：含全图所有 KF（active 窗内可变、外固定），负责找回环 + 误差均摊（直接平均分摊位姿误差，不走 cost flow）+ local BA
- **回环触发**：把老回环帧加入 active window（老帧需 fix），共视点融合
- **后端 Fullgraph**：触发 global BA，结果同步回 real time graph
- **finalba**：原版用于完成时全图优化，耗时极长，可去掉
- **加速方向**：可考虑增量 BA（iSAM/SC-EBA），前提是回环够频繁且非"超大圈才回环"

## 三、本次会议的核心决策

### 决策 1【已定案】Eden 算法路线：OKVIS 为主 + Maplab 兜底

- **以 OKVIS（Fullgraph）为主框架**，目标实时建图 + 任意搬动可恢复 + 多区域无优化耗时（对齐 Terramow 体验）
- **Maplab 保留作为夜间兜底全局优化**（高精度地图重建），不阻塞主路线
- **CPU 平台**：确定继续 X5（3588 黑名单、XX7 贵、全志被制裁）

### 决策 2【已定案】OKVIS Fullgraph 试用路径分两步走

1. **先在 PC 上跑通**官方源码自带的 Fullgraph 流程，用现有数据集评估精度与耗时
2. **若 PC 验证可行**，再上 X5 评估实时性，再考虑工程接入与 Maplab 接口融合

> 注：李宝玉之前为重构改动了大量代码，再把官方 Fullgraph 接回来需要熟练人手。

### 决策 3【已定案】重定位策略

- **建图中搬动 → 必须回到原始位置再继续**（逻辑简单、UX 一致），不学 Terramow"就地继续"
- **多帧重定位校验**：列入计划，用于消除假定位 + 抑制单帧位姿跳变（用一组帧做小优化求更准 pose）
- **当前与老地图挂载频率**：未来若 OKVIS 局部图够稳，可放低到 2–3s/次

### 决策 4【已定案】建图顺序：先弓字再沿边（对齐 Terramow）

- **理由 1**：弓字易频繁回环 → 增量 BA 有优势
- **理由 2**：弓字 IMU 误差可来回抵消，单圈大走误差更大
- **产品对齐**：Terramow / 布查 / Verza 双目机型都自带自主沿边建图能力

### 决策 5【待确认】IMU 选型不再强推 7081

- 现有 LSM6DSV 在李岩的 1h / 8h bias 漂移测试中表现可接受
- 推动 7081 的压力降低；ACC 噪声 / 抗冲击仍是关注点
- **板上 IMU 加热**：可考虑加电阻预加热到工作温度，提升稳定性（参考前公司飞机方案）

### 决策 6【待确认】建图完成后地图持续更新策略

- 思路：用户后续每次大公字割草，按需要合并部分轨迹片段，**后台自动更新地图**（应对季节变化）
- 单次 2 米弓字图约 100MB；完整地图可能 600–700MB，需评估 Flash 容量

## 四、已知问题

### P-1 出桩后 VIO 不与老地图找回环 → 边缘漏割 / 出界

- **症状**：出桩后跑一个新的 VIO/VSLAM，几乎不与已保存地图找回环
- **影响**：边缘漏割、出界
- **修复方向**：高频与老图挂载（已部分在做）

### P-2 RTK 关闭场景下 VIO 漂移严重

- **症状**：杨倩反馈，关 RTK 后摇圈建图，30min 后机器实际已回环，但 APP 上 VIO 已飘到很远，无法点击回环
- **修复方向**：建图时若 RTK 超时 1min 无数据，触发重定位逻辑，不能一直纯信视觉

### P-3 Maplab 未优化地图重定位成功率仅约 20%

- 已有测试结论；提升空间需评估
- 直接拿 Maplab 处理建图中搬动问题不可行

### P-4 Eden 自身 VIO 在场地中央重定位失败 / 跳变

- 与"优化后图在中央区域定位概率较低"现象一致
- **修复方向**：多帧观测校验（决策 3）
