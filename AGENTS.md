## Learned User Preferences

- Always communicate with the user in Chinese (simplified); code and comments in English.
- When running `raw-materials-curation`, always present 「线索摘要」 (phase 1 summary) and wait for explicit user confirmation (often a minimal one-liner like "ok") before proceeding to phase 2 extraction.
- After curation, user may ask follow-up queries about specific documents; search within structured module docs (`decisions.md`, `gaps.md`) first before scanning raw inbox files.
- Weekly tracking 处理：(a) 已完成项简要归纳概括；(b) 待办项保留全部细节及子任务；(c) 长期站点保养（CQIQC / CIIQC 巡线维护等）单独放长期段落，不计入当周 Action items。
- User (宋姝) is the localization team lead and does not personally take on specific execution work items in weekly tracking.
- Daily workspace (`workspace/YYYY-MM-DD/daily.md`) todo section has two subsections: 「跟进/派发」and 「我的工作项」; both are the user's own work. Uncompleted 「跟进/派发」items (unchecked) carry over to the next day's daily.md verbatim until explicitly marked done. All task list items must use markdown to-do syntax `- [ ]` (including nested children), not plain `-`; descriptive notes/group headings remain as plain bullets. Source notes for a new day's `workspace/YYYY-MM-DD/` are usually copied/transformed from `~/工作记录/YYYY-MM-DD/` into the workspace folder.
- MR review ownership: vision team MRs → 宋姝 reviews (goes into 跟进 list); fusion team MRs → 林子越 reviews; laser team MRs → 明坤 reviews. Non-vision MRs are recorded in 团队进展 table only, not in 跟进 list.
- 沟通润色：(a) 对 leadership/PM 的技术程序汇报（如断流）偏好一句话——业务结论先行、关键数字、明确失败兜底（断流口径：「减少断流时停机重定位；6s 内无感恢复定位；失败才执行重定位」），避免 MULLS/setpose 实现术语；(b) 一般向上/横向润色用「先表态配合 → 客观陈述背景与已做工作 → 给明确时间节点 → 留兜底询问」四段式，避免「我们尽量」式被动开头和「不想发有 bug 的版本」式自我怀疑，把视角抬到「为整体节奏/外测/大局考虑」；(c) 飞书同 chat 内给 SPM/产品的更新偏好问句引导式而非硬指令；(d) 向产品/PM 提疑问按「现象」+「疑问」二段式列每个疑问点，疑问语气不质疑追责，金字塔结构、结论先行、口语化但不加废话。
- For single-document curation (one file from `~/下载/`, not a full inbox batch): first output a 「梳理稿」 with 结论先行 + 金字塔结构(论点/论据)，嵌入相关图片预览；只有用户回复「归档」后才真正落盘（复制 inbox、改写 decisions/timeline/gaps/problems、按命名规范 copy 图到 `images/`）。归档后用户常追问 reviewer 式问题（概念澄清、边界 case 挑战），保留刚归档文档的上下文以便就地回答。
- 梳理会议/讨论类文档（非 inbox 标准 curation）时，用户期望结构为 背景 / 需求 / 讨论项 / 工作项；要完整保留「过程中的讨论点和思考点」（决策的隐性 reasoning、被否定的方案、关键反对意见与提出者），而不是只给结论。测试方案整合时偏好按主链路/分支/核验点做层级化梳理，不按"按人分组"的来源罗列；同时避免对原始编号链路做二次改写或自行扩义。
- 工具/方案选型偏好：遇到格式转换、可视化、SLAM/SfM 这类有成熟生态的需求，优先搜现有工具/库（PDAL、PotreeConverter、PCL、COLMAP CLI、glomap 等），不要自行造小脚本/二次魔改；用户多次明确「别自己写脚本，网上搜现有工具」「不要自己改了，查找现有的工具」。
- howto / 操作流程文档（如 `projects/<project>/howto/*.md`）：除最终走通的步骤外，必须把过程中踩过的坑（错误现象 → 根因 → 修复）一并记录，便于复用与避坑。
